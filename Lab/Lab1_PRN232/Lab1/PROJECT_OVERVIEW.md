**Project Overview — ProductManagementASPNETCoreMVC**

This document explains the structure, key code files, how Code-First Entity Framework Core is used, how to run the API, and common commands for development and testing.

**Repository layout**
- **BusinessObjects**: domain models (entities).
  - `AccountMember.cs`, `Category.cs`, `Product.cs` — entity classes with DataAnnotations.
- **DataAccessObjects**: EF Core DbContext and DAOs.
  - `MyStoreContext.cs` — DbContext, OnModelCreating, connection string lookup, and seed data (HasData).
  - `Migrations/` — EF Core migrations generated by `dotnet ef`.
  - DAO classes (e.g., `ProductDAO.cs`, `AccountDAO.cs`) contain direct EF calls used by repository/service layers.
- **Repositories**: repository interfaces and implementations.
  - `IProductRepository.cs`, `ProductRepository.cs` — provide data access abstractions.
- **Services**: business logic layer.
  - `IProductService.cs`, `ProductService.cs` — orchestrate repositories and return models to controllers.
- **ProductWebAPI**: ASP.NET Core Web API project.
  - `Program.cs` — app startup, dependency injection registrations, and middleware.
  - `appsettings.json` — contains connection string `ConnectionStrings:MyStockDB`.
  - `Controllers/ProductsController.cs` — API endpoints for products.

**How data access is wired**
- `MyStoreContext` is the EF Core DbContext. It configures SQL Server using the connection string read from `appsettings.json` at runtime.
- Entity mappings (keys, column names, types) are configured in `OnModelCreating`.
- Seed data was added via `entity.HasData(...)` for `AccountMember`, `Category`, and `Product`.
- The app uses a repository layer (`Repositories`) and a service layer (`Services`) so controllers depend on `IProductService` rather than DbContext directly.

**Migrations & Code-First workflow**
- Add migration:

```bash
dotnet ef migrations add <MigrationName> --project DataAccessObjects --startup-project ProductWebAPI
```

- Apply migration to database:

```bash
dotnet ef database update --project DataAccessObjects --startup-project ProductWebAPI
```

- Remove last migration (if not applied):

```bash
dotnet ef migrations remove --project DataAccessObjects --startup-project ProductWebAPI
```

Note: `--project` is where your DbContext lives (DataAccessObjects). `--startup-project` is where configuration (appsettings, Program.cs) is (ProductWebAPI).

**Seed data**
- Seed data is defined in `MyStoreContext.OnModelCreating` using `HasData(...)`. When you create and apply a migration after adding `HasData`, EF generates a migration that inserts those rows.
- Seeded entities in this project include sample categories, products, and account members.

**Running the API locally**
- Restore packages and build:

```bash
dotnet restore
dotnet build
```

- Run the API (from repo root):

```bash
dotnet run --project ProductWebAPI
```

- By default the app logs show a listening URL, for example: `http://localhost:5230`.

**Common API endpoints (ProductWebAPI)**
- GET all products: `GET /api/Products`
- GET product by id: `GET /api/Products/{id}`
- POST create product: `POST /api/Products` (body: Product JSON)
- PUT update product: `PUT /api/Products/{id}`
- DELETE product: `DELETE /api/Products/{id}`

Controllers return `BusinessObjects.Product` entities; navigation properties may produce nested JSON (be mindful of cycles in serialization).

**Testing quickly with PowerShell (examples used earlier)**

```powershell
# Get products
Invoke-RestMethod -Uri "http://localhost:5230/api/Products"

# Get product 1
Invoke-RestMethod -Uri "http://localhost:5230/api/Products/1"

# Create a product (example)
$body = @{ ProductName = 'Test'; CategoryId = 1; UnitsInStock = 10; UnitPrice = 1.5 } | ConvertTo-Json
Invoke-RestMethod -Uri "http://localhost:5230/api/Products" -Method Post -Body $body -ContentType 'application/json'
```

**Serialization notes**
- Because entities include navigation properties (`Category.Products`), default JSON serialization may include circular references or repeated/null entries. To avoid problems, consider:
  - Return DTOs from the service layer instead of EF entities.
  - Configure JSON options to handle reference loops: e.g., `builder.Services.AddControllers().AddJsonOptions(opt => opt.JsonSerializerOptions.ReferenceHandler = ReferenceHandler.IgnoreCycles);`

**Where to change database connection**
- Edit `ProductWebAPI/appsettings.json` -> `ConnectionStrings:MyStockDB`. The default in this workspace is:

```json
"ConnectionStrings": {
  "MyStockDB": "Server=zawy-laptop;uid=sa;pwd=1234;database=MyStore;TrustServerCertificate=True"
}
```

**Where migrations live**
- `DataAccessObjects/Migrations/` contains migration files, e.g. `20260525045843_InitialCreate.cs`, `20260525050333_SeedMockData.cs`, and the model snapshot `MyStoreContextModelSnapshot.cs`.
